# Мок-кейсы для практики паттернов

Практикум: 29 задач, по одной на паттерн. Каждый кейс — **только ТЗ**, без решения. Реализуешь сам в `src/coding/<категория>/<паттерн>/`.

## Как практиковать (важно)

1. **Сначала напиши `bad`-версию** — в лоб, со `switch`/копипастой/наследованием. Почувствуй боль своими руками.
2. **Потом перепиши через паттерн.**
3. **Критерий успеха везде один:** добавить новый случай = **1 новый класс + 0 правок** в существующем коде (open-closed). Получилось — паттерн применён верно.
4. Каждый кейс заверши классом с `main()` — прогони сценарий, выведи результат.
5. Застрял — подсмотри `good/` соответствующего пакета в основном проекте, но сначала **попробуй сам**.

Темы намеренно отличаются от основного проекта (там fintech) — чтобы это была свежая практика, а не списывание.

Уровень: 🔴 уметь писать с нуля · 🟡 понимать и уметь повторить · ⚪ достаточно узнавать.

## Creational

### 🔴 Singleton 
> Готово (3 варианта: DCL, holder, enum). Добавь 4-й — **настоящий eager**: `private static final INSTANCE = new ...()`, создаётся при загрузке класса безусловно.

### 🔴 Factory Method
> Модуль экспорта отчёта. Пользователь выбирает формат строкой: `"csv"`, `"json"`, `"xml"`. По строке получить экспортёр и вызвать `export(List<Row>)`.

- [ ] Старт: `csv` + `json`. Клиент: `factory.create("csv").export(rows)` — **не зная классов**.
- [ ] Добавить `xml` — правка в одном месте, не в клиенте.
- [ ] Неизвестный формат → понятная ошибка (`unknown format: pdf`).
- [ ] Ноль `if`/`switch` по формату в клиенте.
- [ ] Сделать **двумя способами**: классический (abstract `create()` + подклассы) и registry (`Map<String, Supplier<Exporter>>`).

**Готово когда:** 4-й формат (`yaml`) = 1 класс + 1 строка регистрации, 0 правок в клиенте.

### 🟡 Abstract Factory
> UI-кит под две темы: `LightTheme` и `DarkTheme`. Каждая тема даёт **семейство** согласованных компонентов: `Button`, `Checkbox`, `Scrollbar`.

- [ ] `interface UiFactory { Button createButton(); Checkbox createCheckbox(); ... }`.
- [ ] Клиент получает фабрику и собирает экран, **не зная**, светлая тема или тёмная.
- [ ] Гарантия: нельзя смешать `LightButton` с `DarkCheckbox` (компилятор не даст).
- [ ] Переключение темы = одна строка (`new DarkThemeFactory()`).

**Готово когда:** добавить тему `HighContrast` = 1 фабрика + компоненты, 0 правок в клиенте.

### 🔴 Builder
> Собрать иммутабельный `HttpRequest`. `url` обязателен; `method` (GET по умолчанию), `headers` (мапа), `body`, `timeout` — опциональны.

- [ ] Fluent API: `HttpRequest.builder("http://...").method("POST").header("A","B").build()`.
- [ ] `url` обязателен — без него `build()` кидает исключение.
- [ ] Готовый объект иммутабелен (нет сеттеров, поля `final`).
- [ ] `headers` защити от внешней мутации (`Map.copyOf`).

**Готово когда:** можно собрать и минимальный запрос (только url), и полный — обе читаемы.

### ⚪ Prototype
> Редактор фигур. Есть тяжёлый в создании `Shape` (цвет, координаты, заливка). Нужно «копировать-вставить» — клонировать существующую фигуру вместо создания с нуля.

- [ ] `interface Cloneable { Shape copy(); }`.
- [ ] `copy()` возвращает новый объект с тем же состоянием (deep copy для вложенных).
- [ ] Изменение клона не трогает оригинал — докажи в `main`.

**Готово когда:** клонируешь фигуру, меняешь клон, оригинал не изменился.

## Structural

### 🟡 Adapter
> Твой код работает с интерфейсом `Logger { void log(Level level, String msg); }`. Подключаешь чужую либу `ThirdPartyLog { void write(String jsonLine); }` — менять её нельзя.

- [ ] `ThirdPartyLogAdapter implements Logger`, внутри держит `ThirdPartyLog`.
- [ ] `log(level, msg)` собирает JSON-строку и зовёт `write(...)`.
- [ ] Клиент работает только с `Logger`, про чужую либу не знает.

**Готово когда:** подменив адаптер на другой `Logger`, клиент не меняется.

### ⚪ Bridge
> Уведомления. Две оси: **тип** (`Alert`, `Reminder`) × **канал** (`Email`, `SMS`, `Push`). Наследованием получится 2×3 = 6 классов — взрыв.

- [ ] Канал — интерфейс `Channel { void send(String text); }`.
- [ ] Тип уведомления держит `Channel` как поле и делегирует.
- [ ] Добавить канал `Telegram` — не плодя классы для каждого типа.

**Готово когда:** добавление канала = 1 класс, добавление типа = 1 класс (а не 1×N).

### 🟡 Composite
> Файловая система. `File` (размер) и `Folder` (содержит файлы и другие папки). Нужно `size()` — суммарный размер на любой глубине.

- [ ] Общий интерфейс `FsNode { String name(); long size(); }`.
- [ ] `Folder` держит `List<FsNode>`, `size()` рекурсивно суммирует.
- [ ] Ноль `instanceof` у клиента — он зовёт `node.size()`.

**Готово когда:** `folder.size()` корректно считает вложенную структуру (папка в папке).

### 🔴 Decorator
> Пайплайн обработки текста. Базовый `TextSource.read()` отдаёт строку. Поверх — надстройки: `trim`, `toUpperCase`, `removeDigits`.

- [ ] Все надстройки реализуют `TextSource` и держат внутренний `TextSource`.
- [ ] Комбинируются вложением: `new Trim(new Upper(new Plain("...")))`.
- [ ] Порядок настраивается в рантайме.

**Готово когда:** 3 декоратора дают любую комбинацию без новых классов; добавить `reverse` = 1 класс.

### 🟡 Facade
> Запуск приложения дёргает 4 подсистемы по порядку: `ConfigLoader`, `DbConnector`, `CacheWarmer`, `HealthChecker`. Клиент должен вызвать **один** `start()`.

- [ ] `AppFacade.start()` оркестрирует 4 шага в правильном порядке.
- [ ] Клиент знает только о фасаде, не о подсистемах.
- [ ] Добавить шаг `MetricsRegister` — правка только в фасаде.

**Готово когда:** клиент = одна строка `facade.start()`; порядок шагов закреплён в одном месте.

### ⚪ Flyweight
> Рендер карты с деревьями. 1 000 000 деревьев, но видов всего 3 (`oak`, `pine`, `birch`) — у каждого тяжёлые спрайт+текстура. Хранить миллион копий нельзя.

- [ ] `TreeType` (вид, спрайт) — флайвей, общий. Пул `Map<String, TreeType>` + `private` конструктор + `of(name)`.
- [ ] `Tree` (x, y, ссылка на `TreeType`) — уникальное хранит снаружи.
- [ ] Докажи `==`: два дерева одного вида ссылаются на один `TreeType`.

**Готово когда:** на 1М деревьев создано 3 объекта `TreeType`, не миллион.

### 🔴 Proxy
> Доступ к `ImageLoader { Image load(String path); }`. Реальная загрузка тяжёлая. Нужны: ленивая загрузка (грузить при первом обращении) + кэш.

- [ ] `interface ImageLoader`, реальный `RealImageLoader`, прокси `CachingLazyProxy`.
- [ ] Virtual proxy: реальный объект не создаётся, пока не позвали `load`.
- [ ] Повторный `load` того же пути — из кэша, без реальной загрузки (выведи `cache hit`).

**Готово когда:** первый `load` идёт в реальный загрузчик, второй — нет; клиент не отличает прокси от реального.

## Behavioral

### 🔴 Strategy
> Калькулятор стоимости доставки. Стратегии: `flat` (фикс), `byWeight` (за кг), `freeOver` (бесплатно от суммы X).

- [ ] `interface ShippingStrategy { BigDecimal cost(Order o); }`.
- [ ] Контекст держит стратегию, делегирует ей; стратегия инжектится снаружи.
- [ ] Сменить стратегию в рантайме — без `if/else`.

**Готово когда:** добавить `expressDelivery` = 1 класс; контекст не меняется.

### 🟡 Observer
> `OrderService` меняет статус заказа. На это должны реагировать: отправка email, SMS, обновление склада. Сервис **не должен знать** конкретных слушателей.

- [ ] `interface OrderListener { void onStatusChanged(Order o); }`.
- [ ] Субъект держит `List<OrderListener>`, метод `subscribe`, при изменении обходит всех.
- [ ] Можно подписать/отписать слушателя в рантайме.

**Готово когда:** добавить слушатель `AnalyticsTracker` = 1 класс + подписка; сервис не меняется.

### 🟡 State
> Заказ проходит статусы: `NEW → PAID → SHIPPED → DELIVERED`. Нельзя перепрыгнуть (например, `NEW → SHIPPED` запрещено). Методы: `pay()`, `ship()`, `deliver()`.

- [ ] `interface OrderState { OrderState pay(); OrderState ship(); ... }`.
- [ ] Каждый статус — отдельный класс, разрешает только валидные переходы, остальные → ошибка.
- [ ] Контекст делегирует текущему состоянию, оно возвращает следующее.
- [ ] Ноль `switch(status)`.

**Готово когда:** валидная цепочка проходит, невалидный переход кидает понятную ошибку.

### 🟡 Command
> Пульт умного дома. Кнопки выполняют команды: `LightOn`, `LightOff`, `VolumeUp`. Нужна кнопка `undo` — отмена последней команды.

- [ ] `interface Command { void execute(); void undo(); }`.
- [ ] Команда инкапсулирует получателя + действие.
- [ ] Пульт хранит историю выполненных команд для `undo`.

**Готово когда:** выполнил несколько команд, `undo` откатывает их в обратном порядке.

### ⚪ Chain of Responsibility
> Пайплайн обработки HTTP-запроса: `AuthHandler → RateLimitHandler → ValidationHandler → BusinessHandler`. Любое звено может прервать цепочку.

- [ ] `abstract Handler { Handler next; handle(Request); }`.
- [ ] Звено либо обрабатывает и пробрасывает дальше, либо отклоняет (стоп).
- [ ] Порядок звеньев настраивается при сборке цепочки.

**Готово когда:** валидный запрос проходит все звенья; невалидный отклоняется на нужном.

### ⚪ Template Method
> Парсер отчётов. Скелет одинаков: `open → readLines → parse → close`. Отличается только `parse` для CSV и для TSV.

- [ ] `abstract ReportParser { final parse(file){ open; read; doParse; close; } abstract doParse; }`.
- [ ] `CsvParser` и `TsvParser` переопределяют только `doParse`.
- [ ] Скелет (`final`) переопределить нельзя.

**Готово когда:** добавить `PipeParser` (разделитель `|`) = переопределить 1 метод.

### ⚪ Iterator
> Своя коллекция `RingBuffer<T>` (кольцевой буфер фикс. размера). Нужно обходить её `for-each`, не раскрывая внутренний массив.

- [ ] Реализуй `Iterable<T>` + свой `Iterator<T>`.
- [ ] Обход идёт в правильном порядке (от старого к новому), даже после перезаписи по кругу.
- [ ] Клиент не видит внутренний массив/индексы.

**Готово когда:** `for (T x : ringBuffer)` работает корректно после переполнения.

### ⚪ Mediator
> Чат-комната. Участники (`User`) шлют сообщения, но **не знают друг о друге** — общаются через `ChatRoom`.

- [ ] `Mediator { void send(String msg, User from); }`.
- [ ] `User` держит ссылку на медиатор, не на других юзеров.
- [ ] Медиатор рассылает сообщение всем, кроме отправителя.

**Готово когда:** добавить/убрать участника = работа с медиатором; юзеры друг о друге не знают.

### ⚪ Memento
> Текстовый редактор с `undo`. Состояние — текст + позиция курсора. Нужно сохранять снимки и откатываться.

- [ ] `Memento save()` + `restore(Memento)`; снимок иммутабелен, кишки редактора наружу не торчат.
- [ ] История снимков в отдельном «caretaker».
- [ ] `undo` восстанавливает предыдущее состояние.

**Готово когда:** напечатал, сделал `undo` несколько раз — состояние корректно откатывается.

### ⚪ Interpreter
> Калькулятор булевых правил для фильтра: `age > 18 AND (country == "US" OR vip)`. Распарсить в дерево и вычислить на контексте.

- [ ] `interface Expr { boolean eval(Map<String,Object> ctx); }`.
- [ ] Узлы: `And`, `Or`, `Not`, `GreaterThan`, `Equals`, `Var`.
- [ ] Собери дерево руками (без парсера строки), вычисли на разных контекстах.

**Готово когда:** одно и то же дерево даёт разный результат на разных входных данных.

### ⚪ Visitor
> Дерево AST: `NumberNode`, `AddNode`, `MulNode`. Нужны **две** операции: вычислить значение и напечатать в инфиксной форме — не меняя классы узлов.

- [ ] `interface Visitor { visit(Number); visit(Add); visit(Mul); }`.
- [ ] Узлы: `accept(visitor)` зовёт `visitor.visit(this)`.
- [ ] `EvalVisitor` и `PrintVisitor` — две операции без правки узлов.

**Готово когда:** добавить операцию (`CountNodesVisitor`) = 1 класс, узлы не трогаешь.

## Concurrent

### 🟡 Thread Pool
> Обработать 10 000 задач (каждая «спит» 10мс). Наивно — `new Thread()` на каждую = крах. Нужен пул с лимитом.

- [ ] `ExecutorService` фиксированного размера, `submit` задач, сбор `Future`.
- [ ] Корректное завершение: `shutdown` + `awaitTermination`.
- [ ] Сравни время/память с наивным `new Thread` на каждую задачу.

**Готово когда:** 10к задач обработаны пулом из N потоков без OOM.

### 🟡 Producer-Consumer
> Конвейер: 2 продюсера кладут заказы в очередь, 3 консьюмера разбирают. Скорости разные.

- [ ] Общая `BlockingQueue`, продюсеры `put`, консьюмеры `take`.
- [ ] Корректная остановка (poison pill или флаг + interrupt).
- [ ] Никаких потерянных/задвоенных заказов.

**Готово когда:** все произведённые заказы обработаны ровно один раз, программа завершается чисто.

### 🟡 CompletableFuture (scatter-gather)
> Агрегатор цен: опросить 3 «провайдера» (каждый ~100мс) параллельно, вернуть минимальную цену. Последовательно было бы ~300мс.

- [ ] `supplyAsync` на каждого провайдера.
- [ ] `allOf(...).join()` → собрать все, выбрать минимум.
- [ ] Версия с `anyOf` — вернуть первый ответивший.

**Готово когда:** общее время ≈ времени самого медленного провайдера, не суммы.

### ⚪ Circuit Breaker
> Обёртка над нестабильным сервисом. После 3 ошибок подряд — `OPEN` (фейлим сразу, не дёргаем сервис). Через таймаут — `HALF_OPEN` (пробный вызов).

- [ ] Состояния `CLOSED / OPEN / HALF_OPEN`, счётчик ошибок.
- [ ] В `OPEN` — мгновенный фейл без вызова.
- [ ] Успех в `HALF_OPEN` → `CLOSED`; ошибка → снова `OPEN`.

**Готово когда:** прогон с «падающим» сервисом показывает переходы состояний в правильном порядке.

### ⚪ Retry
> Обёртка с повтором: вызвать ненадёжную операцию, при ошибке повторить с экспоненциальной задержкой (100мс, 200мс, 400мс), максимум 3 попытки, потом fallback.

- [ ] Цикл попыток + нарастающий backoff.
- [ ] После исчерпания — вызвать fallback (или пробросить последнюю ошибку).
- [ ] Успех на 2-й попытке прекращает повторы.

**Готово когда:** операция, падающая 2 раза и успешная на 3-й, отрабатывает; падающая всегда — уходит в fallback.

### ⚪ Read-Write Lock
> Кэш курсов валют: 1000 чтений/сек, 1 запись/сек. Общий лок сериализует чтения — медленно. Нужно: читатели параллельно, писатель эксклюзивно.

- [ ] `ReentrantReadWriteLock`: чтение под `readLock`, запись под `writeLock`.
- [ ] Покажи, что несколько читателей работают одновременно (счётчик активных).
- [ ] Запись блокирует всех.

**Готово когда:** замер показывает параллельные чтения и эксклюзивную запись.
