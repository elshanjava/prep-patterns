# prep-patterns

Шпаргалка по 29 паттернам (GoF + concurrency) на fintech-примерах (платежи, PSP: Stripe / Adyen / Braintree).

Каждый паттерн в проекте лежит в трёх под-пакетах:
- **`bad/`** — наивный код. Открывай первым и спрашивай себя: *что здесь заболит при росте?*
- **`good/`** — тот же сценарий через паттерн.
- **`model/`** — общие доменные классы.

У каждого `*Demo` есть свой `main()` — можно запускать по отдельности. `Main.java` прогоняет всё подряд.
Полный маппинг «паттерн → механизм Spring» с классами и аннотациями — в [`src/SpringMapping.java`](src/SpringMapping.java).

## Как пользоваться шпаргалкой

Читай каждую строку как **боль → приём → механика → реализация → где видел**:
- **Приём** — идея в 5 слов.
- **Механика (рецепт)** — словами: что создаёшь (интерфейс?), что инжектишь, что делегируешь, и какая ООП-техника. Жирным — название приёма.
- **Реализация** — тот же рецепт, но кодом-скелетом на 1 строку. Полный код — в `good/` соответствующего пакета.

Главное наблюдение: почти все паттерны — это **«вынеси меняющуюся часть в интерфейс и держи её как поле (композиция), а не наследуй»**. Исключение — **Template Method** (наоборот, через наследование) и **Singleton** (вообще про контроль конструктора). Если запомнишь это ядро, остальное — вариации «что именно выносим и зачем».

Учи не все 29 сразу. Ядро (80% пользы): **Strategy, Factory Method, Builder, Observer, Adapter, Decorator, Template Method, Singleton**. Остальное — потом.

Метод: прочитал `bad/` → угадал боль → прочитал `good/` → **закрыл и написал `good/` по памяти**. Через день повторил. Чтение даёт ложное «понял»; запоминает только воспроизведение.

## Приёмы под капотом — сводка

Паттернов много, а базовых приёмов — мало. Почти каждый паттерн — это **комбинация 2-3 приёмов отсюда**. Если понять эти приёмы, паттерны перестают быть списком из 29 карточек и становятся вариациями на тему.

| Приём | Что это одной фразой | Зачем | Где применяется (паттерны) |
|---|---|---|---|
| **Композиция** (has-a) | держу другой объект как поле и делегирую ему, а не наследую | заменять/менять поведение, не плодя подклассы | почти везде: Strategy, Bridge, Decorator, Proxy, Adapter, Facade, Composite, Abstract Factory, State, Chain, Command, Thread Pool, Read-Write Lock |
| **Dependency Injection** (DI) | зависимость приходит снаружи (через конструктор), а не `new` внутри | выбирать реализацию в одном месте, легко тестировать (моки) | Strategy, Abstract Factory, Mediator, State, Bridge |
| **Полиморфизм** | работаю через интерфейс, не зная конкретного класса | добавить новый тип — не трогая клиента | Factory Method, Strategy, Chain, и все, где есть интерфейс |
| **Делегирование** | мой метод просто перекладывает работу на хранимый объект | вынести реальную логику в другой объект | Decorator, Proxy, Adapter, Bridge, Facade, State |
| **Наследование** (is-a) | базовый класс фиксирует скелет, наследник переопределяет шаг | переиспользовать общий алгоритм, варьировать детали | **Template Method** (главный), Factory Method (классический вариант) |
| **Оборачивание** (implements тот же интерфейс + держу его) | подменяю объект собой, сохраняя его «лицо» | добавить поведение прозрачно для клиента | Decorator, Proxy |
| **Рекурсивная композиция** | узел держит список детей **того же** интерфейса | единообразно обходить дерево (лист = ветка) | Composite, Interpreter |
| **Инкапсуляция** | прячу внутреннее устройство за узким интерфейсом | клиент не зависит от потрохов, их можно менять | Iterator, Memento, Command, Facade, Singleton |
| **Состояние как объект** | текущий режим — это объект, а не `enum` + `switch` | убрать ветвление, переходы держать рядом с состоянием | State, Circuit Breaker |
| **Реестр / мапа** | `Map<ключ, как-создать/что-делать>` вместо `switch` | новый случай = запись в мапу, а не правка кода | Factory Method (registry-вариант) |
| **Двойная диспетчеризация** | выбор метода по **двум** типам сразу (узел × операция) | добавлять операции, не трогая классы узлов | Visitor |
| **Развязка через буфер** | посредник-очередь между сторонами с разной скоростью | отвязать producer от consumer, сгладить пики | Producer-Consumer |

Как читать: **Strategy** = Композиция + DI + Полиморфизм. **Decorator** = Композиция + Оборачивание + Делегирование. **Composite** = Рекурсивная композиция. Видишь — одни и те же кирпичи в разных сочетаниях.

## Creational — как создавать объекты

| Паттерн | Боль (что в `bad/` ломается) | Приём | Механика (рецепт) | Реализация (скелет) | Где видел (JDK / Spring) |
|---|---|---|---|---|---|
| **Singleton** | объект нужен один на всех (кэш, конфиг), а `new` плодит копии и рассинхрон | один контролируемый экземпляр на процесс | прячешь конструктор (`private`), держишь один статический экземпляр, отдаёшь через `getInstance()`. **Приём: контроль создания** | `private static final INSTANCE`, `private` конструктор, `getInstance()`; проще — `enum` | любой Spring-бин (по умолчанию `@Scope("singleton")`) |
| **Factory Method** | `switch`/`if` по типу разбросан везде, новый тип — правки в 10 местах | вынести создание за одну дверь | создаёшь интерфейс продукта, прячешь `new` за фабричным методом; клиент работает с интерфейсом, не зная класса. **Приём: полиморфизм** | `Map<String, Supplier<T>>` → `registry.get(key).get()`; либо `abstract T create()` в подклассах | `@Bean`-метод в `@Configuration`, `List.of()`, `Calendar.getInstance()` |
| **Abstract Factory** | нужен **согласованный набор** объектов (client+parser одного PSP), легко смешать чужие | фабрика, отдающая когерентное семейство | создаёшь интерфейс-фабрику, инжектишь нужную реализацию в клиента, клиент зовёт `factory.makeX()`. **Приём: композиция + DI** | `interface Factory { Client makeClient(); Parser makeParser(); }` + одна реализация на семейство | `@Configuration` + `@Profile("prod"/"test")` подменяет всё семейство бинов |
| **Builder** | конструктор на 8 параметров, половина опц.; `new X(null, 0, null, true...)` нечитаем | пошаговая сборка с понятными именами | выносишь сборку в отдельный объект: копишь поля сеттерами, в конце `build()`. **Приём: пошаговая композиция** | `class Builder { setX(){…; return this;} … T build(); }`; каждый сеттер возвращает `this` | `StringBuilder`, `UriComponentsBuilder`, `WebClient.builder()`, Lombok `@Builder` |
| **Prototype** | создать объект дорого (тяжёлая инициализация), а нужно много похожих | клонировать готовый шаблон | создаёшь интерфейс с `copy()`, держишь эталон, клонируешь его вместо `new`. **Приём: делегирование клонирования объекту** | `interface Prototype { T copy(); }`; `copy()` возвращает `new` с тем же состоянием | `@Scope("prototype")`, `ObjectProvider<T>`, `Object.clone()` |

## Structural — как складывать объекты вместе

| Паттерн | Боль | Приём | Механика (рецепт) | Реализация (скелет) | Где видел |
|---|---|---|---|---|---|
| **Adapter** | чужой класс с несовместимым интерфейсом, менять его нельзя | обёртка-переходник к нужному интерфейсу | создаёшь целевой интерфейс, инжектишь чужой объект внутрь адаптера, метод переводит вызов. **Приём: композиция (оборачивание)** | `class Adapter implements Target { Adaptee a; void req(){ a.specificMethod(); } }` | `HandlerAdapter` (MVC), `JpaVendorAdapter`, `Arrays.asList()` |
| **Bridge** | две оси изменений (тип × реализация) дают комбинаторный взрыв классов | разделить абстракцию и реализацию | инжектишь реализацию (интерфейс) в абстракцию и делегируешь ей; обе оси растут отдельно. **Приём: композиция вместо наследования** | `abstract Abstraction { Impl impl; }` — методы делегируют в `impl`; оси меняются независимо | Spring Data `Repository` ↔ Jpa/Mongo, `PlatformTransactionManager` |
| **Composite** | клиент по-разному обрабатывает «один объект» и «группу» | дерево, где лист и ветка — один интерфейс | общий интерфейс; узел инжектит список детей того же типа и рекурсивно им делегирует. **Приём: рекурсивная композиция** | `Composite implements Node { List<Node> kids; exec(){ kids.forEach(Node::exec); } }` | `ProviderManager` (список `AuthenticationProvider`), `CompositeMeterRegistry` |
| **Decorator** | нужно добавлять поведение (лог, кэш, retry), наследование даёт взрыв подклассов | обернуть объект, сохранив интерфейс | реализуешь тот же интерфейс, инжектишь оборачиваемый объект, добавляешь до/после и делегируешь. **Приём: композиция** | `class Dec implements C { C inner; op(){ extra(); inner.op(); } }`; обёртки вкладываются | `BufferedReader(new FileReader(...))`; `@Transactional`/`@Cacheable`/`@Async` (AOP) |
| **Facade** | подсистема из 5 классов с хрупким порядком вызовов торчит наружу | один простой вход, прячущий сложность | инжектишь подсистемы как поля, в одном методе вызываешь их по порядку. **Приём: композиция (оркестрация)** | `class Facade { a; b; c; pay(){ a.x(); b.y(); c.z(); } }` — клиент зовёт один метод | `JdbcTemplate`, `RestClient`, сам `ApplicationContext` |
| **Flyweight** | миллионы похожих объектов жрут память дублями | разделить неизменяемую часть между всеми | инжектишь кэш-мапу, отдаёшь общий immutable-объект вместо нового. **Приём: разделение ссылок** | `Map<Key, Obj> cache`; `get(k)` возвращает общий immutable-объект, не `new` | пул бинов-синглтонов, `ConversionService`, Hibernate 2nd-level cache |
| **Proxy** | нужен контроль доступа/ленивость/удалённость, не трогая сам объект | подставка с тем же интерфейсом | реализуешь тот же интерфейс, инжектишь реальный объект, добавляешь проверку/ленивость и делегируешь. **Приём: композиция (контроль доступа)** | `class Proxy implements S { S real; op(){ checkAccess(); real.op(); } }` | Spring AOP (`@Transactional`), `@FeignClient`, Spring Data Repository, Hibernate lazy |

## Behavioral — как объекты взаимодействуют

| Паттерн | Боль | Приём | Механика (рецепт) | Реализация (скелет) | Где видел |
|---|---|---|---|---|---|
| **Chain of Responsibility** | один метод проверяет всё (fraud, лимит, гео) — растущий ком из `if` | цепочка звеньев, каждый решает «обработать / передать дальше» | создаёшь интерфейс-обработчик; каждый инжектит ссылку на следующего и передаёт дальше. **Приём: композиция (список) + полиморфизм** | `abstract Handler { Handler next; handle(r){ if(stop) …; else next.handle(r); } }` | `SecurityFilterChain`, `HandlerInterceptor`, servlet-фильтры |
| **Command** | вызов = операция + аргументы + undo + очередь, всё намешано в сервисе | упаковать запрос в объект | создаёшь интерфейс `Command`, кладёшь в объект данные+получателя; инжектишь команды в исполнитель, он зовёт `execute()`. **Приём: инкапсуляция вызова** | `interface Command { execute(); }`; команда хранит receiver+аргументы, исполнитель зовёт `execute()` | `ApplicationEvent` + `@EventListener`, `@Async`, `@Scheduled`, Spring Batch Job |
| **Interpreter** | правила/выражения захардкожены в Java, бизнес не может менять | грамматика + дерево, вычисляющее выражение | создаёшь интерфейс-выражение; узлы инжектят под-выражения и рекурсивно их вычисляют. **Приём: рекурсивная композиция** | `interface Expr { boolean eval(ctx); }`; `And`/`Or`/`Not` — узлы, держащие под-выражения | SpEL (`@PreAuthorize`), JPQL `@Query`, `@Value("${...}")` |
| **Iterator** | клиент лезет во внутреннее устройство коллекции, чтобы её обойти | единый способ обхода без раскрытия структуры | коллекция отдаёт объект-итератор; клиент крутит `hasNext()/next()`, не зная устройства. **Приём: инкапсуляция обхода** | `interface Iterator { hasNext(); next(); }`; коллекция отдаёт его через `iterator()` | `Iterator`/`for-each`, Spring Data `Page`/`Slice`, `RowMapper` над `ResultSet` |
| **Mediator** | N объектов знают друг о друге напрямую → связность N² | посредник, через которого все общаются | создаёшь интерфейс-медиатор, инжектишь его в компоненты; они зовут медиатор вместо друг друга. **Приём: централизация через DI** | `Mediator { notify(sender, event); }`; компоненты держат ссылку на медиатор, не друг на друга | `ApplicationContext`, `ApplicationEventPublisher`, Spring Cloud Gateway |
| **Memento** | нужно откатить состояние, но не вскрывая инкапсуляцию объекта | снимок состояния во внешнем хранителе | объект отдаёт immutable-снимок наружу и умеет восстановиться из него. **Приём: инкапсуляция состояния** | `Memento save()` + `restore(Memento)`; `Memento` — immutable-снимок полей | `@Transactional` savepoint/rollback, Hibernate dirty-checking, Spring Session |
| **Observer** | при изменении надо уведомить многих, а издатель жёстко знает всех получателей | подписка: издатель шлёт событие, не зная подписчиков | создаёшь интерфейс-наблюдатель; субъект держит список и подписывает их, при событии всех обходит. **Приём: композиция (список) + инверсия зависимости** | `Subject { List<Observer> subs; notify(){ subs.forEach(o->o.update(e)); } }` | `publishEvent()` + `@EventListener`, Micrometer, `ApplicationListener` |
| **State** | поведение зависит от статуса → гигантский `switch(status)` в каждом методе | состояние = объект, переходы внутри него | создаёшь интерфейс-состояние; контекст инжектит текущее и делегирует ему, состояние само переключает ссылку. **Приём: композиция вместо `switch`** | `interface State { handle(ctx); }`; каждое состояние само вызывает `ctx.setState(next)` | Spring Statemachine, Spring Batch `JobStatus`, `SecurityContext` |
| **Strategy** | алгоритм выбирается рантайм-условием → `if/else` по типу везде | вынести меняющийся алгоритм в интерфейс | создаёшь интерфейс стратегии, инжектишь реализацию в контекст, контекст делегирует ей. **Приём: композиция + DI** | `interface Strategy { exec(); }`; контекст держит `Strategy` и делегирует ему | `Comparator`, любой `@FunctionalInterface`, `@Autowired List<FeeStrategy>` |
| **Template Method** | у нескольких процессов одинаковый скелет, отличается пара шагов | базовый алгоритм фиксирован, шаги переопределяемы | наоборот: базовый класс держит алгоритм, наследник переопределяет шаг-хук. **Приём: наследование (единственный тут)** | `abstract { final run(){ step1(); hook(); } abstract hook(); }`; подкласс меняет шаг | `JdbcTemplate.query()` (свой `RowMapper`), `AbstractHealthIndicator` |
| **Visitor** | новая операция над деревом объектов = правки во всех классах узлов | вынести операцию в отдельный «посетитель» | инжектишь визитор в `accept()`, узел зовёт `visitor.visit(this)`. **Приём: двойная диспетчеризация** | `interface Visitor { visit(A); visit(B); }`; узел: `accept(v){ v.visit(this); }` (double dispatch) | `BeanDefinitionVisitor`, обход AST в SpEL, `JsonNode.traverse()` |

## Concurrent — многопоточность

| Паттерн | Боль | Приём | Механика (рецепт) | Реализация (скелет) | Где видел |
|---|---|---|---|---|---|
| **Thread Pool** | `new Thread()` на каждую задачу → 1000 потоков → OOM, нет контроля | переиспользуемый пул с лимитом | инжектишь `ExecutorService` как поле, сабмитишь задачи в него вместо `new Thread`. **Приём: композиция (пул внутри сервиса)** | `var pool = Executors.newFixedThreadPool(n); pool.submit(task)`; в конце `pool.shutdown()` | `@Async` + `ThreadPoolTaskExecutor`, `Executors.newFixedThreadPool()` |
| **Producer-Consumer** | производитель и потребитель работают с разной скоростью, прямой вызов блокирует | буфер-очередь между ними (backpressure) | инжектишь общую `BlockingQueue` обоим; producer кладёт, consumer берёт. **Приём: развязка через общий буфер** | `BlockingQueue q`; producer `q.put(x)`, consumer `q.take()` — очередь сама блокирует | `@KafkaListener`/`@RabbitListener`, `BlockingQueue`, Spring Integration Channel |
| **CompletableFuture (scatter-gather)** | 3 PSP опрашиваются последовательно → 300мс вместо 100мс | разослать параллельно, собрать результат | запускаешь N задач асинхронно, комбинируешь их future через `allOf/anyOf`. **Приём: композиция future** | `supplyAsync(...)` ×N → `allOf(...).join()` (все) или `anyOf(...)` (первый) | `@Async` → `CompletableFuture`, `allOf`/`anyOf`, WebFlux `Mono.zip()` |
| **Circuit Breaker** | внешний сервис лёг, а мы долбим его и копим зависшие потоки → каскадный отказ | «предохранитель»: после N ошибок OPEN → fail-fast | оборачиваешь вызов в объект со счётчиком ошибок и состоянием; в `OPEN` бросаешь сразу. **Приём: обёртка + state** | счётчик ошибок + state `CLOSED/OPEN/HALF_OPEN`; в `OPEN` бросаем сразу, по таймауту пробуем `HALF_OPEN` | Resilience4j `@CircuitBreaker` + `fallbackMethod`, Spring Cloud Circuit Breaker |
| **Retry** | разовый сетевой сбой роняет операцию, хотя повтор бы прошёл | повтор с backoff, потом fallback | оборачиваешь вызов в цикл с нарастающей паузой, после N попыток — fallback. **Приём: обёртка** | `for(attempt) try return call(); catch { sleep(backoff*2^n); }`; после N → fallback | `@Retryable` + `@Backoff` + `@Recover`, `RetryTemplate` |
| **Read-Write Lock** | read-heavy кэш: общий лок сериализует даже параллельные чтения | много читателей параллельно, писатель эксклюзивно | инжектишь лок как поле, оборачиваешь чтение в `readLock`, запись в `writeLock`. **Приём: композиция (лок внутри)** | `var l = new ReentrantReadWriteLock()`; read → `l.readLock()`, write → `l.writeLock()` | `ReadWriteLock`/`StampedLock`, `@Version` (optimistic), `@Cacheable`/`@CacheEvict` |

## Ресурсы

- **[Refactoring.Guru](https://refactoring.guru/ru/design-patterns)** — есть русская версия, визуализации «проблема → решение». Лучший старт.
- **Head First Design Patterns** — самая «незабываемая» подача, через истории. Берёт именно когда «не запоминается».
- **[Baeldung](https://www.baeldung.com/)** — короткие статьи «паттерн X в Java» с реальным кодом.
- **Christopher Okhravi** (YouTube) — объясняет голосом *зачем*, а не *как*.
- **GoF «Design Patterns»** — первоисточник, но академичный. Как справочник, не для старта.
